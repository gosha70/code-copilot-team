### B.4 Proposed commit message (draft)

```
refactor(benchmark): remove vllm backend stub — providers ≠ backends

Phase 2c+2d registered ``vllm`` as a backend stub with a Phase-3-deferred
``NotImplementedError``. The premise — that vLLM is a peer to Claude Code
in the backend taxonomy — was wrong. vLLM is a *provider* (an LLM-serving
HTTP runtime that speaks OpenAI Chat Completions or Anthropic Messages),
not an agentic *backend* (the thing that drives an editor loop).

The corrected architecture (see specs/benchmark-harness/audit-2026-05-08.md):

- Backends are copilot agents: ``claude-code``, ``aider``, ``codex``,
  ``github-copilot`` (and ``stub`` for CI).
- Providers are HTTP endpoints, configured per-copilot via the copilot's
  own env vars (``ANTHROPIC_BASE_URL`` for Claude Code, ``OPENAI_API_BASE``
  for Aider, ``COPILOT_PROVIDER_BASE_URL`` for GH Copilot CLI,
  ``[model_providers.<id>]`` in ``~/.codex/config.toml`` for Codex).
- The harness *records* which provider a run used by reading the env vars
  at run time and writing them into ``backend_metadata``. It does NOT set
  them — provider configuration is the user's responsibility (or, for the
  forthcoming standalone provider-config feature in
  specs/provider-config/, CCT's responsibility through a provider-profile
  emitter that extends shared/templates/provider-profile-template.toml).

Removed:
- scripts/benchmark_runner/backends/vllm.py (stub)
- scripts/benchmark_runner/tests/test_vllm_backend.py
- vllm registration in scripts/benchmark_runner/_register.py
- benchmarks/README.md sections referencing vllm as a backend

Test count drops from 138 to ~131 (the 7 vllm-stub tests are removed
along with the stub).

Refs: #32 (rescope)
```

## Open question for you

**`shared/templates/provider-profile-template.toml` extension scope.** The existing schema is rich (4 types, healthchecks, per-type optional fields) but its `command` field is shaped for peer review (`{review_request}`, `{model}` placeholders). Two ways to extend it for benchmarks:

- **Option 1 (minimal):** Don't add new fields. The harness reads provider type + base_url + model + api_key_env from the existing schema and constructs its own invocation per backend. The peer-review-specific `command` field is ignored by the harness. Adapters like Codex's TOML config get emitted by the standalone provider-config feature based on the same TOML.

- **Option 2 (per-use-case extensions):** Add an `[providers.<id>.benchmark]` subsection with fields like `temperature`, `seed`, `extra_headers`, etc. Schema gets richer; both peer review and benchmark stay declarative.

I'll write the `specs/provider-config/spec.md` after you indicate which direction. If you want to defer the standalone-feature spec until after Phase 3 is reshaped, I can stub it as a placeholder for now.
