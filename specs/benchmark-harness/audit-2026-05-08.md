Per directive: re-read the working spec line by line, identify every design point that was derived from the wrong premise (harness calls the LLM directly, vLLM is a peer to claude-code as a benchmark "backend"), and replace each with the verified-fact alternative.

**Fact base:** [`doc_internal/copilot-llm-support-matrix.md`](../../doc_internal/copilot-llm-support-matrix.md). Every replacement below cites the row of that matrix it relies on.

The corrected branch's Phase 2c+2d commit ships the worktree+venv tier and dogfood subset cleanly, without the vllm stub. Backups frozen under `_backup-2026-05-08-pre-correction/` capture the pre-correction spec/plan/tasks for diff convenience; the pre-rebase commit-level state is reachable via the reflog if needed.

## Spec.md — line-pinned issues

### L43 — "leaderboard for `claude-code:sonnet` we can use as a sanity reference"

**Current text (Strategic framing):**
> The MVP ships exactly one public adapter — Aider Polyglot — because it has a published leaderboard for `claude-code:sonnet` we can use as a sanity reference …

**Issue:** Aider's published Polyglot leaderboard reports scores for **Aider-the-agent driving model X**. There is no published score for *Claude Code driving Claude Sonnet on Polyglot*. Comparing our `claude-code:sonnet` numbers to Aider's leaderboard is comparing two different agents (Claude Code's tool loop vs Aider's edit-block loop) running the same underlying model — not "validating our harness against a known result."

**Correction:** Reframe the dogfood gate. Either (a) compare *our Aider backend* numbers to Aider's leaderboard (apples-to-apples, requires adding an Aider backend), or (b) drop the leaderboard-comparison framing entirely and treat dogfood as a "the harness runs end-to-end without crashing on a real public benchmark" gate. Recommendation: **drop the leaderboard comparison**, keep the run-to-completion gate.

### L53–L54 — User scenario 1 frames `claude-code:sonnet` and `vllm:<model>` as peers

**Current text:**
>    `./scripts/benchmark run --benchmark aider-polyglot --backend claude-code:sonnet --runs 3`
>    and again with `--backend vllm:<model> --runs 3`. They then run
>    `./scripts/benchmark report --run-dir runs/<ts>/`.

**Issue:** This presents `vllm:<model>` and `claude-code:sonnet` as peer backends. They are not. `claude-code:<model>` drives a copilot agent (Claude Code's tool loop, file edits, multi-step reasoning); `vllm:<model>` (as drafted) drives a raw Chat Completions call with no agent loop. Comparing them measures *different things* (full agent vs bare model), not "two backends on the same task."

**Correction:** Backends are **copilot agents**: `claude-code`, `aider`, `codex`, `github-copilot` (and `stub`). LLM/provider choice is a **separate axis** — set externally via env vars the copilot itself reads (`ANTHROPIC_BASE_URL`, `OPENAI_API_BASE`, `COPILOT_PROVIDER_BASE_URL`, `[model_providers.<id>]` in `~/.codex/config.toml`). The peer-comparison scenario becomes: same backend, different providers. Or: same provider, different backends.

Matrix-row support: see `doc_internal/copilot-llm-support-matrix.md` § per-copilot rows for each.

### L83–L85 — User scenario 5: "Local vLLM run for an open-weights model"

**Current text:**
> 5. **Local vLLM run for an open-weights model.** A maintainer with a vLLM
>    server running locally exports `CCT_VLLM_ENDPOINT=http://localhost:8000/v1`
>    and runs `./scripts/benchmark run --benchmark aider-polyglot --backend vllm:<model> --runs 1`.

**Issue:** Same abstraction-layer error. The realistic version of this scenario is: maintainer has a vLLM server, points Claude Code (or Aider, or Codex, or GH Copilot CLI) at it via that copilot's own gateway env vars, then runs the benchmark with the **copilot** as the backend.

**Correction (concrete rewrite):**
> 5. **Local-LLM run via Claude Code gateway.** A maintainer launches a vLLM server with `--enable-auto-tool-choice --tool-call-parser openai`, exports `ANTHROPIC_BASE_URL=http://localhost:8000`, `ANTHROPIC_AUTH_TOKEN=dummy`, `ANTHROPIC_DEFAULT_SONNET_MODEL=<served-model-name>`, and runs `./scripts/benchmark run --benchmark aider-polyglot --backend claude-code:<served-model-name> --runs 1`. Claude Code routes its model calls to the local vLLM, the harness measures the full copilot+model stack.

Matrix support: Claude Code row (vLLM integration), `docs.vllm.ai/en/stable/serving/integrations/claude_code/`.

### L132–L133 — Backend scope (MVP)

**Current text:**
> - `claude-code:<model>` — invokes `claude -p` headlessly, streams transcript to disk, parses usage from the final transcript event. …
> - `vllm:<model>` — OpenAI-compatible HTTP client against a configured endpoint. Captures token usage where present in the response. …

**Issue:** This explicitly conflates two different roles into the same "Backend" abstraction. `claude-code:<model>` is a copilot (Claude Code) using a model. `vllm:<model>` is a raw HTTP client to a model server. Different abstraction layers.

**Correction:** Replace the backend list with copilot agents only:
- `stub` — copies golden patch into worktree (CI smoke).
- `claude-code:<model>` — `claude -p` headless. Provider routing via `ANTHROPIC_BASE_URL` (default Anthropic API; can point at vLLM/Ollama/LM Studio gateway).
- `aider:<model>` — `aider --message` non-interactive (Phase 4 candidate; pending verification of headless flag).
- `codex:<model>` — `codex exec` headless (Phase 4 candidate).
- `github-copilot:<model>` — `copilot -p` headless with BYOK env vars (Phase 4 candidate).

**Out of scope** — Cursor, Windsurf (GUI-only, no headless agent loop — see matrix's "Headless invocation" rows).

### L376 — Risks: vLLM endpoint variability

**Current text:**

**Issue:** This is now a Claude-Code-via-gateway concern, not a direct-vLLM-client concern. The risk is real but moves location.

## What stays unchanged

The architectural correction does NOT touch any of these:

- Adapter contract (`BenchmarkAdapter`, `IsolationConfig`, `isolation_for`) — still right.
- Aider Polyglot adapter — unchanged. It's an adapter, not a backend, and the polyglot tasks are agnostic to which copilot drives them.
- Stub adapter, stub backend — unchanged.
- Run orchestration (`run.py`) — minor: the `backend.metadata` already carries arbitrary keys, so capturing the gateway env vars there is additive, not breaking.
- Run-record schema (`run-record.json`, `score.json`, `stats.json`) — unchanged. The `backend_invocation.model` field already carries the model identifier; we add `provider` (or `endpoint`) to `backend_metadata` for gateway-routed runs.
- All committed phases (0, 1, 2a–2d) on `feat/benchmark-harness` — none of them carry the wrong-abstraction code; they all assume backend = copilot agent.
- The polyglot `python_version` / `install_command` flow — unchanged.
- Worktree+venv tier, isolation tier resolution — unchanged.

So the rollback is **uncommitted-only**: no committed phase reverts. The rewrite touches the spec/plan/tasks/README and the uncommitted Phase 3 drafts.
